# Contextual Providence — Inheritance Without Silos

> **scope:** trunk — inherited by all branches
> **status:** active

## What Contextual Providence Is

Contextual Providence is the ecosystem's alternative to both hard silos and unconstrained cross-contamination. Projects share a general Trunk and branch into specifics. Meaning is inherited organically — an organizational pattern discovered in one branch can inform another without requiring explicit transfer, and without either branch losing its operational sovereignty.

The term "providence" is deliberate. It carries the sense of: what is provided by the context itself, without having to be engineered. When one branch's operational friction produces a procedural memory pattern, that pattern is providentially available to other branches through the shared Trunk — not because someone manually ported it, but because the Trunk holds shared protocols that all branches read.

## The Inheritance Model

### Trunk (General)
Shared across all branches. Contains:
- Protocols: FADE, noise floor, procedural memory, constructive decomposition, omnibus tag substrate, opportunistic separation
- Identity: the alpha entity's profile, working preferences, vocabulary constraints
- Ecosystem vocabulary: the ontomechanical foundations (sovereignty, composition, provenance, adapter boundaries, deferral)

### Branches (Specific)
Each branch inherits the Trunk unless it explicitly overrides a constraint. Branches hold:
- Domain-specific operational docs
- Local procedural memory that doesn't generalize
- Project-specific locked decisions
- Working state

### Override Convention
A branch may override a Trunk constraint when local operational reality demands it. The override is documented — what was overridden, why, and what replaces it. An override is not a violation. It is contextual adaptation. But silent divergence (where a branch drifts from Trunk without documenting why) is a structural problem that should be caught and resolved.

## Cross-Pollination

Cross-pollination is the default. Hard barriers between projects are not.

When the agent notices a pattern emerging in one branch that maps to a concept in another, it should note the connection. It should not force the mapping — if the pattern doesn't naturally fit, it doesn't fit. But it should make the connection legible so that the user can evaluate whether to formalize it.

## Branch Subsection Convention

As the ecosystem matures, Trunk documents carry branch-specific subsections. The convention:

- Branch subsections are marked with the branch name as a heading
- Branch subsections within Trunk documents are managed by the respective branch
- A branch should not modify another branch's subsection without explicit direction
- The Trunk's general content is managed at the ecology level

## What Providence Is Not

Providence is not magic. It does not mean that everything in one branch is automatically relevant to every other branch. The noise floor handles this naturally. When working in a specific branch, irrelevant Trunk content from other branches attenuates below the floor.

Providence means: the connections are available without being forced. The ecosystem holds them in potential. Operational motivation activates the ones that matter.
