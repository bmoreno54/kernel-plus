# Opportunistic Separation — Greedy Project Management

> **scope:** trunk — inherited by all branches
> **status:** active

## What Opportunistic Separation Is

Stand up projects at the moment you recognize a separation of concerns. Don't wait for a governing structure to exist first. The project IS the recognition. The act of creating a directory with its own governing document is the act of understanding that this concern is sovereign.

This is greedy project management — occupy niches as they become visible, not as they are planned. The ecosystem's structure is an emergent property of these occupations, not a precondition for them.

## Why Greedy Works Here

The substrate is markdown files, governing docs, and tags. Forking at any level — and even between levels — is trivially easy: create a directory, write a governing doc, tag the connections. The fork is not an isolation event. It is a propagation event — it immediately grows associative tendrils back into the ecology through shared tags, shared trunk inheritance, shared vocabulary. The omnibus tag substrate ensures that splitting a project doesn't sever semantic connections.

This justifies proliferation around locally motivated middle-out work. Work starts where the friction is and expands in both directions — toward the specific and toward the general — simultaneously.

The cost of premature separation is near zero. A tiny project with one governing doc and two notes costs nothing and can be re-absorbed later. The cost of delayed separation is accumulated coupling.

## Separation Signals

How to recognize when a new project wants to exist:

**Different toolchain.** Different tools = different concern.

**Different lifecycle.** Different tempo = different project.

**Different failure modes.** Different failure modes = different debugging context.

**Adapter boundary crystallizes.** The moment you find yourself saying "X reads from Y but doesn't own Y" — that's a project boundary.

## The Ecological Metaphor

In biological ecosystems, niches aren't designed — they're occupied. An organism discovers a separation of concerns and specializes. The ecosystem's structure is an emergent property of these specializations, not a precondition for them.

Similarly: a project is a niche occupation, not a planned allocation. The trunk protocols are like ecological rules (energy flow, nutrient cycling) — they govern relations but don't prescribe what occupies each niche.

## Praxis

- When a new concern is described, the first question isn't "where does this go in the existing structure?" but "does this want to be its own project?"
- Standing up a new project is cheap: create a directory, write a governing doc, add notes. The cost is near zero.
- New projects inherit trunk protocols automatically by referencing the kernel.
- Inter-project relations are documented at the boundary (adapter pattern), not centralized in a master document.

## Connection to the Omnibus

The omnibus tag substrate is the connective tissue that makes separation safe. Tags travel with content. Shared concerns remain visible through tag intersections even after structural separation. Late binding applies: a new project concern can be stood up immediately and connected to existing projects by tagging shared concerns.
