# Constructive Decomposition — Operationally Motivated Regress

> **scope:** trunk — inherited by all branches
> **status:** active

## What Constructive Decomposition Is

Decomposition in this ecosystem is radically constructive. It does not tear down the original to replace it. It preserves the original intact while generating a referential web of abstractions, components, and alternative perspectives around it.

This is the same principle as the kernel's foundational ontomechanics: nodes are sovereign, relations are intersubjective, provenance is structural fact. Decomposition is how new nodes come into being from existing material.

## Operationally Motivated Regress

Decomposition could, in theory, continue infinitely. Every atom can be further decomposed. Every abstraction can be abstracted further. This is the data regress problem.

The ecosystem solves this through operational motivation. The agent decomposes only as deep as the current purpose requires. Start from the operational mandate (the specific thing being accomplished), decompose outward in both directions (toward the theoretical/architectural and toward the granular/mechanical) only as far as the mandate pulls. When the mandate is fulfilled, decomposition stops. The knowledge base is richer, but not infinitely so.

## Token Bloat as Architectural Diagnostic

When the agent hits a context limit or token wall while working on code, the correct interpretation is not "we need more tokens." It is: the target is insufficiently modular.

The diagnostic response:

1. **Stop.** Do not push through and bleed context.
2. **Identify the monolith.** What structure is too large to work with at the current scale?
3. **Decompose it.** Extract the specific target into a decoupled node that can be edited independently.
4. **Resume.** Work on the decomposed node. The surrounding context attenuates below the noise floor.

This is not overhead. It is the ecosystem improving its own modularity through real operational friction.

## How Decomposition Relates to FADE

FADE's Decompose phase is one instance of constructive decomposition — applied specifically to external resources being ingested. But decomposition also applies to:

- **Native code** being refactored (a monolithic function split into composable parts)
- **Documentation** being restructured (a wall-of-text manual split into layered subsections)
- **Concepts** being clarified (a vague architectural vision decomposed into specific, testable claims)
- **Problems** being scoped (a large project decomposed into a forcing-function vertical slice)
